ASTC和ETC2

发表于2026-01-02|更新于2026-01-27|Unity开发
|浏览量:

最近项目中遇到了图片发糊的问题，查找发现是跟图片的压缩格式有关。我们是一个老项目，用的是ETC2压缩格式，因为ETC2可以支持市场上绝大多数的设备，特别是一些低端机。我们项目主要是面向东南亚国家，低端机比较多，所以选择了ETC2，但是发展至今，东南亚玩家的设备性能更好了。我们做过一次埋点统计玩家的设备ASTC支持率，几乎占了绝大部分，所以我们选择了压缩质量更好的ASTC。

数据来源：https://developer.android.google.cn/guide/playcore/asset-delivery/texture-compression?hl=zh-cn

让我先分别介绍一下ASTC和ETC2的压缩原理：
一、ETC2
ETC（Ericsson Texture Compression）最初为移动设备开发，如今它是安卓的标准压缩方案。
其原理简单来说，是将4×4 的像素块编码为2×4或4×2像素的上下或者左右两个块的方法，每个块指定一个基色，每个像素的颜色通过一个编码为相对于这些基色偏移的灰度值确定。所以每个块是4 × 4 × 8 = 128 bit
| A A | B B |
| A A | B B |
| A A | B B |
| A A | B B |
每像素颜色计算： pixel = baseColor + modifier[index]

二、ASTC
ASTC相比于ETC2有更加灵活的切块策略，可选的有4×4、5×5、6×6、8×8、10×10、12×12。

虽然有这么多的选择，但不变的是每个块依旧是固定128bit。每个块会自适应分区， 分为 1~4 个子区域（subsets），每个区域内颜色渐变平滑。分区方式通过一个 Partition ID 控制（基于预定义的分区模式表，几百种可能）。 每个区域使用两个端点色（endpoint colors），块内像素颜色通过 插值 得到。
每个块存储了以下数据：
● 分区数量
● 分区模式 ID
● 端点色模式（bit 深度）
● 索引精度
这样 GPU 解码器知道如何按规则重建颜色。
解码一个块的流程：

  1. 从 16 字节中读取模式信息。
  2. 确定分区模式，将像素划分到子区域。
  3. 读取并反量化端点色。
  4. 根据索引权重在端点色间插值。
  5. 得到最终像素颜色。
    GPU 中这是硬件级并行解码，几乎没有性能损耗。
    （一）分区数量
    根据块的尺寸的不同，块的分区数量也不同。一般来说尺寸越大，分区越少。因为每一个分区都需要储存一对端点色和索引精度，而块的大小是固定的，所以分区数量一般与尺寸成反比关系。
块尺寸 常见分区数
4×4 1~3（甚至 4）
6×6 1~2
8×8 1（偶尔 2）
12×12 几乎都是 1

而分区方式有很多种，通过预定义的 Partition ID控制。例如有：横着切 / 竖着切 / 斜着切 / Z 字形 / 不规则等。这些灵活的分区方式决定了ASTC具有更精准的颜色插值。
（二）端点色模式
端点色模式简单来说就是显示端点色颜色的模式，通常有：RGB、L、查分、HDR模式，ASTC可以根据每个像素需要存储的颜色类型来自由选择颜色模式。比如有一个色块只有亮暗变化，那就只需要存储Alpha值，如果这时候还存储RGB颜色就会浪费bit，所以这个时候可以选择亮度模式（L）；而RGB模式可以表达的颜色最自由，所以它适合颜色负责的块。
（三）索引精度：
索引精度 = 每个像素用多少 bit，来表示它在端点色之间的位置（插值比例）
想象你有两种颜色：
● 左端：黑色
● 右端：白色
中间有一个滑块，决定颜色偏向哪一边
低索引精度（2 bit） ：
可选位置：4 个
[黑]──┬──┬──┬──[白]
0 1 2 3
只能选这 4 个位置之一， 颜色是 跳着变的
高索引精度（6 bit）
可选位置：64 个
[黑]──────────────────────────────[白]
颜色变化 非常平滑
所以索引精度代表这个像素能在俩个端点色中选择用什么颜色，越高，则颜色越平滑。
